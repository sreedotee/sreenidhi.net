Statewatch

It's 4am. Something's broken in the checkout I built last weekend, and the bug isn't in the screens I drew. It's in a state nobody drew. Designers ship state bugs now, so I went and learned a 40-year-old technique people kept writing about. Tried it on a checkout flow. With Alloy actually verifying it.

Tool
State machines, Alloy, Stately
Year
2026
Type
Play
Topic
Behavioral spec

Half my design work happens in code now. So do half my bugs. Not the kind where a margin breaks in Safari. The kind where the user ends up in a state I never drew. A modal that won't close because the payment is stuck in some retry loop the spec didn't anticipate. A button that stays loading after the request finished. The Figma file said one thing, the code drifted into another, and neither of us noticed.

The brief I started withI'm designing a standard e-commerce checkout flow. The customer has items in their cart, reviews them, enters payment details, completes payment, and gets a confirmation. I need to handle the common edge cases properly: - Declined cards (the card is rejected by the bank) - 3D Secure challenges (the bank wants extra verification) - Network failures during payment - Sessions that expire mid-checkout - Items going out of stock between adding to cart and paying - The user navigating back at various points Don't lose state if they refresh. Don't let them double-charge if they double-click. The whole thing should feel robust. I want this modeled as a state machine so I can see all the states laid out, find the holes in my own design before I ship, and use it as the source of truth my code references.

State machines were invented for exactly this kind of thing. Harel published the formalism in 1987. Horrocks wrote a whole book applying it to UI in 1999. Khourshid built XState. Hillel Wayne formally verified a real shipped UI (Edmodo Snapshot) with Alloy in 2018. sketch.systems and Stately Studio ship the "state machine with frames attached" pattern as products today. None of this is new. It just hasn't caught on with designers, even though it's a designer's problem.

The diagram came out of about an hour of work. AI drafted the machine from the brief in XState. Then I ran Alloy on it. Alloy doesn't find bugs on its own — it verifies properties you specify. I encoded the brief's requirements as Alloy assertions ("in-flight payments survive session expiry", "success states cannot regress", "stock changes handled in payment states") and asked the solver to check them.

Alloy verification — first pass00. check allStatesReachable UNSAT ✓ holds 01. check noDeadlock UNSAT ✓ holds 02. check inFlightSurvivesSessionExpiry SAT ✗ counterexample 03. check successCannotRegress SAT ✗ counterexample 04. check stockHandledEverywhere SAT ✗ counterexample 05. check networkTimeoutHandled SAT ✗ counterexample 4 of 6 properties failed. Each failure = an Alloy counterexample showing a sequence of events that violates the assertion. In plain language: ✗ User in mid-3DS, session timer expires, machine kicks them to "session lost." Order may already be charged. Brief said "in-flight is sacred." ✗ User on the receipt screen, session expires, machine drops them off the receipt to "session lost." Order is done; this shouldn't matter. ✗ Inventory webhook fires while user is mid-payment. Machine has no defined behavior. Brief said handle stock changes "between adding to cart and paying." ✗ User in NetworkError, another timeout fires. Silently dropped. Designer call: visible counter or genuinely ignore?

Fixed each one. Re-ran Alloy. Every property held. The verified machine has 14 atomic states across cart, coupon, payment, and outcome clusters, plus two parallel regions for session and network. Most of those states weren't in the initial AI draft — they came out of fixing what Alloy flagged. That's the workflow's first dividend: 4 real bugs caught before any UI code was written.

The second dividend is maintenance. I added a gift-coupon feature to the verified machine — three new states, seven new events, transitions into the existing flow from the Reviewing screen. Re-ran Alloy. Round trip was about fifteen minutes.

Alloy verification — after adding gift coupon// the 6 previously-verified invariants — re-checked 00. check allStatesReachable UNSAT ✓ still holds 01. check noDeadlock UNSAT ✓ still holds 02. check inFlightSurvivesSessionExpiry UNSAT ✓ still holds 03. check successCannotRegress UNSAT ✓ still holds 04. check stockHandledEverywhere UNSAT ✓ still holds 05. check networkTimeoutHandled UNSAT ✓ still holds // new invariants written for the coupon feature 06. check couponHandlesSessionExpiry SAT ✗ counterexample 07. check couponValidationHandlesTimeout SAT ✗ counterexample 08. check couponFlowExits UNSAT ✓ holds Extension didn't regress any existing invariant. Two new bugs found in the coupon feature itself: ✗ Coupon entry/validation states don't handle session expiry ✗ Coupon validation (server call) doesn't handle network timeout Both are the cross-cutting concerns a designer focused on feature happy paths always misses. Caught in seconds.

The 11 states in the verified machine map to 11 screens. Not wireframes — not comps — screens that exist because the machine exists. When you add a state, you add a frame. When Alloy rejects a state, no frame gets built for it.

Why this matters. When someone adds a feature, they extend the machine and re-run Alloy. The previously-verified invariants protect the verified core — if the extension breaks something already proven, Alloy says so. Anyone on the team can ask the spec questions: how many happy paths exist, can a user reach this state without verifying their email, what's the shortest path to an error. The chart is interrogable. States and their UI frames live in the same artifact — when the flow changes, the diagram changes, the verification re-runs, the frames are re-attached. Stale documentation is the eternal problem. This one stays current because it is the truth, not derivative of it.

The machine isn't the runtime. Production code stays whatever you build it as — vibecode, real React, legacy Rails, doesn't matter. The spec is a separate document that intent lives in: formally checkable, visually documented, read by engineers to know what to build, used by tests (auto-generated via xstate-test) to catch drift. Khourshid spells this out in his CSS-Tricks piece on model-based testing. Wayne demonstrated the verification angle on the Edmodo Snapshot app in 2018. sketch.systems and Stately ship the state-machine-with-attached-frames pattern. The piece that isn't yet a product is AI doing the manual labor those tools currently leave to the designer — drafting the machine from intent, generating frames per state. That's what I tried here.

This is a play exercise, not production-validated. One brief, one flow, one extension, on my own machine. The AI translation steps need real review under real conditions. Stately's Figma embed feature is paid. The maintenance pattern needs to hold up across many features and team rotations, not a single addition. But each piece of the chain — Harel's formalism, Horrocks' maintained-spec pattern, Khourshid's tooling, Wayne's verification approach, Lucas's sketch.systems, Stately's embed — has been published and used. Composing them, with AI in the loop, held together once. Whether it scales is the next question.
